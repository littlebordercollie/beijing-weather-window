# MVP 规格：雨窗

版本：0.1
目标：2—3 天内验证“目的地 + 到达时间”的降雨决策是否比普通天气页更有用。

## 1. 核心用户问题

用户不需要先知道站名，也不需要学会读雷达图。第一版只回答：

> 我准备去某个地方，预计某个时间到。现在走会不会淋？如果等一会儿，什么时候走更合适？

示例输入：

```text
我现在从望京出发，18:40 到朝阳公园南门，骑车会不会淋？
```

示例输出结构：

```text
现在出发：中风险
预计 18:28—18:42 路上有 8—12 分钟小到中雨。
18:50 后出发更稳；目的地西南侧雨区正在接近。
置信度：中。
```

这只是界面文案样例，不是一次真实天气计算结果。

## 2. 第一版范围

### 必须有

- 输入地点；
- 输入预计到达时间，交通方式可选；
- 地点解析为坐标；
- 目的地当前天气：温度、气压、湿度、过去一小时降水、风、观测时间；
- 目的地未来两小时、每 5 分钟降水时间线；
- 目的地中心 + 东西南北约 3 km 的四个格点探针；
- 当前官方预警；
- 现在走、晚 10/20/30 分钟的风险对比；
- 每条数据展示来源、数据类型和更新时间；
- 每 5 分钟保存一次原始响应和计算结果。

### 第三天有余力再加

- 过去两小时雷达动画；
- 简单路线采样；
- “过去回波持续多久”的保守描述；
- 可展开的计算证据卡。

### 第一版明确不做

- 不训练天气大模型；
- 不宣称拥有北京全部官方气象站；
- 不把四个格点称为四个气象站；
- 不用网页内部未文档化接口作为正式核心数据源；
- 不给雷阵雨输出单一、确定到分钟的停雨承诺；
- 不在未确认许可前长期批量抓取水务历史或公开发布雷达瓦片。

## 3. 页面结构

### 首页

1. 一句话输入框；
2. 地点、到达时间和交通方式的解析确认；
3. 主结论卡：风险、建议出发窗口、置信度；
4. 三个备选方案：现在、晚 10 分钟、晚 30 分钟；
5. 数据更新时间与预警提示。

### 证据抽屉

- 当前实况卡；
- 两小时降水时间线；
- 五探针小地图；
- 雷达动画；
- 数据来源与冲突说明；
- “为什么这样判断”的可读解释。

### 以后再加的“北京观测地图”

- 国家站层；
- 水务日雨量站层；
- 获得授权后的区域自动站层；
- 雷达层；
- 时间轴与逐站历史曲线。

## 4. 数据架构

```text
用户自然语言
  -> 结构化地点、出发/到达时间、交通方式
  -> 高德 POI / 路线
  -> 坐标统一为中国大陆服务所需的 GCJ-02
  -> 目的地 + 四个周边探针
  -> 和风实况、分钟降水、预警
  -> 可选 RainViewer 历史雷达（WGS-84 / Web Mercator）
  -> 统一数据模型
  -> 确定性雨窗计算器
  -> AI 解释器
  -> 页面
```

### 坐标注意事项

- 高德和和风在中国大陆使用 GCJ-02；
- RainViewer 雷达瓦片基于 WGS-84/Web Mercator；
- 叠加前必须显式转换，不能把坐标系差异当成“雷达偏了”。

## 5. 统一数据模型

所有观测和预测都必须带证据字段：

```json
{
  "source": "qweather",
  "data_kind": "grid_forecast",
  "measure": "precipitation",
  "location": {"lon": 116.0, "lat": 40.0, "crs": "GCJ-02"},
  "valid_at": "2026-07-11T18:40:00+08:00",
  "observed_or_generated_at": "2026-07-11T18:20:00+08:00",
  "received_at": "2026-07-11T18:21:02+08:00",
  "value": 0.4,
  "unit": "mm/5min",
  "quality": null,
  "license_tag": "qweather-attribution-required",
  "raw_snapshot_id": "..."
}
```

允许的 `data_kind` 至少包括：

- `station_observation`：真实多要素站；
- `rain_gauge_observation`：真实雨量站；
- `grid_analysis`：融合实况；
- `grid_forecast`：格点预报；
- `radar_echo`：雷达回波；
- `official_warning`：官方预警；
- `derived_decision`：程序计算出的行动建议。

## 6. 确定性计算逻辑

AI 不直接算雨量和时间。程序完成以下计算，AI 只把结果说清楚。

### 单点雨窗

对未来两小时每 5 分钟降水序列：

1. 使用配置阈值把每个时间片分为无雨/小雨/中雨/较强；
2. 合并连续降雨区间；
3. 计算首次降雨、最后降雨和峰值；
4. 寻找至少连续 15 分钟的低风险窗口；
5. 在 ETA 前后各 10 分钟计算目的地风险。

阈值必须集中配置，并在 UI 帮助页说明；不能让大模型临时决定。

### 五探针一致度

- 中心点为目的地；
- 东、西、南、北各约 3 km；
- 比较五点开始/结束时间和强度变化；
- 至少三个点显示一致的先后顺序，才用“雨区可能向某方向移动”；
- 若变化顺序不稳定，输出“局地生成/消散明显，移动方向不稳定”。

这里的方向只是格点预报的辅助解释。真正的空间运动以连续雷达帧为主。

### 路线风险

路线版把路线按预计经过时间采样：

```text
路线风险 = 各采样点在预计经过时刻前后 10 分钟内的最大降雨风险
```

不能只查路线沿途“现在”的天气，因为用户到达每个点的时间不同。

### 置信度

置信度由程序根据以下因素计算：

- 数据新鲜度；
- 五探针一致度；
- 分钟预报和自研雷达外推是否一致；
- 最近雷达回波是否快速增强/新生；
- 是否有雷电或强对流预警；
- 是否存在缺测。

不同来源冲突超过阈值时，输出一个区间或明确“来源判断不一致”，不能强行取平均。

## 7. “已经下了多久”的口径

若只有过去两小时、10 分钟一帧的雷达：

```text
雷达显示该位置上空自 14:10 左右开始出现降水回波，
回波已持续约 30—40 分钟。
```

不能写“地面已经精确下了 37 分钟”，原因包括：

- 雷达帧有时间间隔；
- 空中回波不等于雨滴已落到地面；
- 回波会受距离、地形、杂波和衰减影响。

获得分钟雨量站数据或从产品上线后持续自建记录，才能更可靠地回答地面持续时间。

## 8. 三天开发拆分

### 第 1 天：数据链路

- 创建高德和和风项目；
- 地点解析为 GCJ-02 坐标；
- 跑通实况、分钟降水和预警；
- 生成五探针；
- 建统一数据模型；
- 原始 JSON 落 SQLite；
- 验证北京相距数公里的点是否返回不同分钟序列。

### 第 2 天：决策界面

- 一句话输入和结构化确认；
- 主结论卡；
- 两小时降水时间线；
- 五探针地图；
- 现在/晚 10/20/30 分钟对比；
- 计算规则与置信度；
- 全部卡片显示来源和时间戳。

### 第 3 天：雷达与验收

- 加过去两小时雷达动画；
- 若时间允许，接路线采样；
- 增加来源冲突提示；
- 选择一次真实降雨过程，保存原始 JSON、请求时间、预测结论和后续实况；
- 形成可复现验收报告。

## 9. 调用量粗算

一次手动刷新最低约：

- 5 次分钟降水；
- 1 次实况；
- 1 次预警。

共约 7 次。若每 5 分钟刷新，一小时约 84 次。服务端必须按“数据源 + 格点 + 更新时间窗口”缓存和去重，不能让每个浏览器直接打供应商接口。

正式额度和价格必须以创建项目后控制台为准。

## 10. 验收标准

### 功能验收

- 任意北京 POI 能解析到坐标；
- 输入 ETA 后能返回现在、晚 10/20/30 分钟三组方案；
- 五探针数据可展开；
- 每个数值都有来源、数据类型和时间；
- 供应商失败或数据过期时明确显示失败，不用旧值伪装当前值；
- 同一份输入和原始响应能复现同一风险结果。

### 事实边界验收

- 页面不出现“附近四个气象站”，除非数据确实带真实站号和站坐标；
- 雷达回波持续时间不写成地面降雨持续时间；
- 预警原文和官方预报不被 AI 改写为相反结论；
- AI 不生成任何接口中不存在的数值；
- 所有冲突和缺测都有可见状态。

### 真实天气验收

至少记录一次真实降雨过程：

- 首次预测的雨开始/结束区间；
- 每次更新的原始响应；
- 后续实况或可用观测；
- 开始时间误差、结束时间误差和漏报/空报；
- 对用户建议是否实际有帮助的人工复盘。

## 11. 升级路线

1. 路线时空采样；
2. 彩云或其他有授权的未来雷达/闪电数据；
3. 申请北京区域自动站站表和分钟数据；
4. 用真实站点校准雷达估雨；
5. 积累预测—实况对照，展示不同地区/天气类型的历史误差；
6. 数据足够后再考虑 pySTEPS 光流、概率集合或本地深度临近模型。
